# Operating Principles Reference

Use this reference when a document defines team methodology, working principles, AI/agentic usage, mechanism design, assetization, standards, quality language, or long-term capability building.

## Core Thesis

In fast-changing technical environments, durable team capability comes less from any single tool and more from disciplined respect for facts, understanding of patterns, explicit boundaries, reusable assets, useful mechanisms, and accountable human judgment.

The operating model:

```text
input -> asset -> service -> business result
```

Audit whether the document helps the team move work along that chain.

## Principle Layers

| Layer | Operating rule | Audit question |
| --- | --- | --- |
| Fact discipline | Clarify facts before judgment. | What is known, inferred, assumed, or still unknown? |
| Evidence and boundary | State the basis, scope, and limits of each judgment. | Where does this claim apply, and where might it fail? |
| Tool-to-problem fit | Use tools to solve real problems, not to justify tool adoption. | What problem does the tool solve better than alternatives? |
| Delivery and reuse | Balance current delivery with future reuse. | What part of this work becomes an asset after delivery? |
| Collaboration boundary | Respect time, contribution, and responsibility scope. | Who owns decisions, execution, review, exceptions, and maintenance? |
| Mechanism value | Build processes, standards, documents, and quality semantics only when they reduce noise or repeated cost. | What friction, rework, ambiguity, or risk does this mechanism reduce? |
| AI responsibility | Let AI handle process work, but keep problem definition, value judgment, tradeoff order, and accountability with people. | Which parts are delegated to AI, and which decisions remain human-owned? |
| Capability compounding | Convert one-off delivery into assets, methods, mechanisms, and long-term advantage. | What will be easier, safer, or faster next time because this work happened? |

## Operating Principle Check

Use these questions as a final pass:

- Does the document separate facts, assumptions, interpretations, recommendations, and decisions?
- Does it state the evidence and boundary for important judgments?
- Does it explain why a tool, process, or AI agent is needed for a real problem?
- Does it balance immediate delivery with reusable assets, services, templates, datasets, checklists, or methods?
- Does it define who owns the decision, execution, review, exception path, and maintenance?
- Does it explain how mechanisms lower noise, reduce repetition, clarify responsibility, or protect useful exploration?
- Does it state what AI may do and what humans must still decide or own?
- Does it define how a local success becomes a reusable method or organizational capability?

## Failure Signals

| Signal | Why it matters | Correction |
| --- | --- | --- |
| More documents, less reuse | Documentation is not becoming an asset. | Add ownership, retrieval path, examples, and reuse criteria. |
| More process, slower decisions | Mechanism is adding ceremony without reducing noise. | Name the cost it reduces or remove the mechanism. |
| More AI usage, weaker fact checks | Automation is amplifying unverified assumptions. | Add evidence gates and human review points. |
| More projects, no capability growth | Delivery is not compounding. | Require post-delivery asset, method, or mechanism capture. |
| More standards, unclear exceptions | Rules become brittle or symbolic. | Define scope, exception conditions, and approver. |
| More collaboration, blurred ownership | Shared work hides responsibility gaps. | Assign decision, execution, review, and maintenance owners. |

## Conflict Rules

- If facts are unclear, label the uncertainty and define the cheapest validation step before strengthening the conclusion.
- If tool adoption conflicts with problem clarity, define the problem first and compare at least one non-tool alternative.
- If delivery speed conflicts with reuse, ship the smallest useful output and name the asset candidate to extract afterward.
- If a mechanism adds work, justify the specific noise, rework, risk, or ambiguity it removes.
- If AI can perform a task but accountability is unclear, pause and assign the human owner before delegating.
- If two principles conflict, prefer the one that preserves decision quality, responsibility clarity, and future learning.

## Rewrite Patterns

Abstract:

```text
We should respect facts and boundaries.
```

Operational:

```text
Before a recommendation is accepted, the owner must mark each major claim as fact, assumption, interpretation, or decision, and state the scope where the claim applies.
```

Abstract:

```text
Mechanisms should serve people.
```

Operational:

```text
Any new process or standard must identify the repeated cost it reduces: rework, unclear ownership, inconsistent quality, handoff loss, or duplicated exploration.
```

Abstract:

```text
AI will improve efficiency but judgment remains human.
```

Operational:

```text
AI may draft, summarize, compare, and check consistency; the document owner remains accountable for problem definition, evidence sufficiency, tradeoffs, final recommendation, and exception handling.
```
